# 医疗器械全生命周期追溯管理系统需求文档

## 1. 文档目的

本文档基于《医疗器械全生命周期追溯管理系统设计与实现》开题报告整理，用于明确毕业设计原型系统的业务目标、功能范围、数据要求、非功能要求和本次实现边界，作为系统设计、编码实现和测试验收的依据。

## 2. 项目背景

医疗器械监管正在从单点记录转向全生命周期可追溯。传统纸质台账和人工记录在来源核验、流向跟踪、责任界定、召回处置和维护校准等场景中存在明显短板。为满足 UDI 制度推进与医院精细化管理的需求，系统需要构建覆盖采购登记、仓储流转、临床使用、维护校准、追溯查询等环节的闭环管理能力。

## 3. 建设目标

- 建立统一的医疗器械主数据中心，实现 UDI、院内编码、供应商编码、RFID、二维码等多码关联。
- 支撑器械从入库到使用、维护、校准、报废前状态管理的全流程记录。
- 实现按器械、批次、患者、科室等多维度的正向和逆向追溯。
- 为毕业设计提供一个可直接演示的软件原型和可复用的论文实现基础。

## 4. 业务范围

本次原型聚焦医院内医疗器械管理，不覆盖真实采购审批流、财务对账、第三方监管平台真实接口和正式权限认证，只实现演示所需的核心业务闭环。

业务范围包括：

- 器械主数据登记
- 多码映射解析
- 入库、出库、批量盘点
- 患者使用绑定
- 维护、维修、计量校准登记
- 预警提示
- 全维度追溯查询

## 5. 用户角色

- 系统管理员：维护基础资料，查看全局数据。
- 库房管理员：执行入库、出库、盘点操作。
- 临床护士或耗材管理员：登记器械在患者和科室中的使用情况。
- 设备工程师：维护设备保养、维修、校准记录。
- 管理人员：查看追溯链、预警与统计信息。

原型系统不实现复杂 RBAC，仅在需求层保留角色划分。

## 6. 核心业务流程

### 6.1 主数据建档

1. 管理员录入器械基本信息、分类、规格型号、供应商、生产厂商、批次和效期。
2. 系统为器械保存 UDI 及院内编码，同时可绑定供应商编码、RFID、二维码等外部标识。
3. 系统生成首条追溯事件，作为生命周期起点。

### 6.2 仓储流转

1. 库房管理员通过 UDI、RFID 或二维码识别器械。
2. 系统登记入库、出库或盘点动作。
3. 系统自动变更库存数量、位置、状态，并记录追溯事件。

### 6.3 临床使用

1. 临床人员扫描器械标识并选择患者、科室和手术信息。
2. 系统建立“器械-患者-科室-操作人-时间”关联关系。
3. 系统沉淀临床使用追溯记录，用于后续召回和责任追踪。

### 6.4 维护校准

1. 设备工程师录入保养、维修或校准计划和结果。
2. 系统维护下次到期时间并生成预警。
3. 系统将维护活动写入追溯时间轴。

### 6.5 追溯查询

1. 用户输入 UDI、RFID、二维码、批次号、患者编号或患者姓名。
2. 系统解析查询对象并返回对应追溯视图。
3. 支持查看时间轴、影响患者范围、关键码映射和当前位置。

## 7. 功能需求

### FR-01 基础信息中心

- 支持新增器械主数据。
- 支持维护器械名称、分类、风险等级、规格型号、供应商、生产厂商、批次号、序列号、单位、效期、补货阈值等信息。
- 支持展示供应商、科室、患者等基础资料。
- 支持查看每个器械的当前状态、库存和位置。

### FR-02 多码映射管理

- 系统必须支持 UDI、院内编码、供应商编码、RFID、二维码五类标识。
- 同一器械允许绑定多个编码，但每个编码值在系统内必须唯一。
- 用户在任意业务环节输入任意一种编码时，系统都应能解析到同一器械对象。

### FR-03 智能仓储管理

- 支持按单个标识完成入库。
- 支持按单个标识完成出库。
- 支持一次录入多个标识完成批量盘点模拟。
- 系统应记录数量、位置、操作人、备注和时间。
- 系统应在库存不足时阻止非法出库。

### FR-04 临床使用追溯

- 支持登记患者编号、患者姓名、科室、手术名称、操作人、使用数量和备注。
- 支持将器械与患者绑定，形成追溯链。
- 临床使用后系统应自动更新库存和器械状态。

### FR-05 维护校准管理

- 支持登记保养、维修、校准三类记录。
- 支持维护计划日期、完成日期、执行人、服务单位、结果和下次到期时间。
- 支持查询历史维护记录。
- 支持输出即将到期维护预警。

### FR-06 预警管理

- 支持效期预警。
- 支持维护校准到期预警。
- 支持低库存预警。

### FR-07 全维度追溯门户

- 支持按编码查询单个器械的全生命周期视图。
- 支持按批次号查询关联器械和受影响患者范围。
- 支持按患者编号或姓名查询该患者使用过的器械。
- 支持按时间轴展示关键追溯事件。

### FR-08 仪表盘

- 支持展示器械总数、即将过期数、维护到期数、服务患者数、低库存数等指标。
- 支持展示近期预警列表。

## 8. 数据需求

### 8.1 核心实体

- 器械：器械主数据、状态、库存、位置、效期等。
- 编码映射：UDI、院内编码、供应商编码、RFID、二维码。
- 供应商：名称、资质编号、联系人、联系电话。
- 科室：名称、类型。
- 患者：编号、姓名、性别、年龄。
- 库存流水：入库、出库、盘点事件。
- 临床使用记录：患者、器械、科室、手术、数量、时间。
- 维护记录：类型、计划、结果、下次到期时间。
- 追溯事件：生命周期时间轴记录。

### 8.2 数据约束

- 编码值唯一。
- 库存数量不得为负。
- 临床使用数量必须大于 0。
- 维护记录必须关联具体器械。
- 追溯事件必须保留时间、动作、位置和操作者信息。

## 9. 非功能需求

### 9.1 易用性

- 页面需具备清晰的信息分区和表单提示。
- 关键功能应在三步以内完成操作。

### 9.2 可追溯性

- 所有关键业务动作必须同步写入追溯事件。
- 查询结果需以时间轴形式直观呈现。

### 9.3 性能

- 演示环境下常规查询响应时间应控制在 2 秒内。
- 支持批量盘点模拟时一次处理多条编码输入。

### 9.4 可维护性

- 前后端应模块化组织。
- 数据库设计需便于扩展后续报废、召回、监管接口等功能。

### 9.5 安全性

- 原型阶段仅实现基础数据校验和输入约束。
- 正式系统需补充登录认证、权限控制、审计与脱敏能力。

## 10. 本次原型实现边界

本次项目实现以下内容：

- Web 端可运行演示系统
- SQLite 本地数据库
- 内置演示数据
- 基础信息、仓储、临床使用、维护校准、追溯查询、预警与仪表盘
- 采购申请、审批、到货入库闭环
- 召回管理与报废管理
- 登录与角色权限控制
- 审计日志与 Excel 报表导出
- 正向路径、逆向影响和流程图关系追溯视图

本次不实现以下内容：

- 对接真实 RFID 读写器硬件
- 对接国家或企业级 UDI 平台
- 真实扫码枪接入
- 财务管理、采购审批、报废审批
- 多角色登录权限与第三方单点认证

## 11. 验收标准

- 系统可启动并打开 Web 页面。
- 可新增器械主数据并完成多码绑定。
- 可完成入库、出库、临床使用、维护登记。
- 可通过编码、批次号、患者编号完成追溯查询。
- 可正确输出预警信息和仪表盘统计。
- 核心业务逻辑具备自动化测试支撑。

## 12. 建议迭代方向

- 接入正式权限模型和审计日志。
- 增加召回管理、报废管理、统计报表导出。
- 接入真实扫码设备、RFID 中间件和消息队列。
- 增加监管平台接口与区块链存证能力。

## 13. 本轮扩展实现说明

针对原型阶段之后的优先功能，本轮已补充以下能力：

- `采购到入库闭环`：支持采购申请、审批、到货入库，并自动沉淀追溯事件。
- `召回管理`：支持按批次或单器械发起召回、统计影响器械和患者、关闭召回流程。
- `报废管理`：支持报废申请、审批与最终处置，并自动联动库存状态。
- `权限控制`：提供管理员、库房管理员、临床护士、设备工程师、管理人员五类演示角色。
- `审计日志`：所有关键写操作统一记录用户、动作、目标和时间。
- `统计报表`：支持导出 Excel 综合报表，覆盖库存、临床、维护、采购、召回、报废和审计。
- `增强预警`：新增召回预警、超期未校准预警和长期未盘点预警。
- `增强追溯`：在原时间轴基础上补充正向路径、逆向影响和流程图关系视图。
